Semantic retrieval using Sentence-BERT (all-MiniLM-L6-v2) with cosine similarity over dense embeddings
Phase 1 — Data Preparation
The first thing to do is make sure all three datasets are clean and ready. For GoEmotions, load the trimmed CSV and verify that the text column has no nulls, no duplicate rows, and that the primary_label_name column is correct. For EmpatheticDialogues, check that the prompt and response column pairs are complete. For Spotify Lyrics, confirm the mood or genre tag column is consistent across all rows.
After that, run text preprocessing on the GoEmotions text column and the EmpatheticDialogues text. This includes converting all text to lowercase, removing URLs, special characters, and excessive whitespace, removing stopwords, and applying lemmatization. The Spotify lyrics dataset does not need heavy preprocessing at this stage since it is used for retrieval, not model training.
For EmpatheticDialogues, check that the prompt and response column pairs are complete with no nulls. Since the retrieval system will use Sentence-BERT instead of TF-IDF, the preprocessing requirements are different. Sentence-BERT encodes full sentences into dense semantic embeddings, which means it benefits from natural, readable text rather than heavily stripped text. For this dataset, use clean_text as the input column — lowercased with punctuation and special characters removed, but with stopwords kept and grammatical structure preserved. Do not use final_text for Sentence-BERT, as stopword removal and aggressive lemmatization degrade the semantic quality of the sentence embeddings. The final_text column in EmpatheticDialogues can be ignored entirely for this pipeline.

Phase 2 — Emotion Detection Model Fine-Tuning
With clean data ready, the next step is fine-tuning the emotion detection model. Load the pretrained RoBERTa base model from HuggingFace using the transformers library. Tokenize the GoEmotions text column using the RoBERTa tokenizer with padding and truncation. Split the trimmed dataset into train (80%), validation (10%), and test (10%) sets.
Fine-tune the model using the Trainer API from HuggingFace or a manual PyTorch training loop. Train for 3 to 5 epochs with a learning rate around 2e-5 and a batch size of 16. Monitor validation loss at each epoch to avoid overfitting.
After training, evaluate the model on the test split using accuracy, precision, recall, and F1-score per emotion class. Generate a confusion matrix across all 28 emotion labels. Save the final model weights and tokenizer locally so it can be loaded by the Flask API later.

Phase 3 — Retrieval-Based Response System
Phase 3 implements a retrieval-based empathetic response system using the cleaned training_pairs_clean.csv and response_bank_clean.json files stored in the Kaggle dataset. The system loads a pretrained Sentence-BERT model, all-MiniLM-L6-v2, using the sentence-transformers library and encodes all user_message entries from training_pairs_clean.csv into dense vector embeddings. These embeddings are stored as a NumPy matrix and saved as a .npy file so they do not need to be recomputed each time the API starts. When a new user message is received, the same model encodes the input into an embedding, and cosine similarity is computed between the input embedding and all stored user_message embeddings. To improve retrieval quality, the detected emotion from the classifier is first mapped from the five output labels (sadness, anger, joy, anxiety, and calm) into related emotion groups found in the cleaned dataset, and similarity search is limited to that emotion bucket. The system then returns the paired assistant_reply associated with the most similar user_message. If the similarity score is below a defined threshold or no suitable emotion group is available, the system falls back to response_bank_clean.json to select a safe empathetic response from the corresponding emotion category. After retrieval, the selected reply is passed through a response-composition layer that adds a short supportive follow-up sentence and a brief music-recommendation statement based on the detected emotion, producing the final empathetic chatbot response. This module is evaluated using standalone test sentences and is designed to be integrated directly into the Flask API in the next phase.

Phase 4 — Mood-to-Music Mapping Logic
In parallel with Phase 3, build the mood-to-music mapping logic. Using the emotion label output from the model, map it to a valence and arousal score based on the mapping table defined in the project context. Then translate that valence-arousal position into a mood category such as melancholic, intense, euphoric, tense, or peaceful.
Connect to the Spotify API using the Spotify Python library. Set up OAuth credentials and test playlist search by mood category and genre. Build a function that takes a mood category as input and returns a list of 5 songs with title, artist, genre, mood tag, Spotify URL, and cover image URL.

Phase 5 — Flask API Development
With the NLP model, retrieval system, and music mapping all working independently, the next step is integrating everything into a Flask API. Set up a Flask project structure with a main app.py file. Create two endpoints: POST /api/chat which handles the full pipeline from user message to response and playlist, and GET /api/playlist which can be called separately if needed. Inside the /api/chat endpoint, the flow is as follows: receive and validate the JSON request body, extract the message field, pass it through the text preprocessing function, run it through the fine-tuned RoBERTa model to get the dominant emotion label and confidence score, pass the original clean_text to the Sentence-BERT retrieval system by encoding it into an embedding and computing cosine similarity against the preloaded response embedding matrix to get the empathetic response text, map the emotion label to a mood category and call the Spotify API to get the song list, then package everything into the response JSON structure and return HTTP 200. Add basic error handling for missing fields, text that is too short, and API failures. Add a Flask session or MongoDB connection for storing chat history per user. Test all endpoints thoroughly using Postman before handing off to the frontend developer.

Phase 6 — Frontend Development
Once the Flask API is stable and tested, begin building the frontend using Vite React and Tailwind CSS. Set up the Vite project and install Axios for HTTP requests.
Build the chat interface with a text input area, a send button, and a scrollable message history panel. When the user submits a message, show a loading spinner, send an Axios POST request to /api/chat, then render the empathetic support text as a chat bubble and the playlist as a row of song cards below it. Each song card should display the cover image, song title, artist name, mood tag, and a clickable link to open the track on Spotify.
Handle error states by showing a friendly message if the API returns a 4xx or 5xx response. Make sure the input is disabled while a request is in progress and re-enabled after the response renders.

Phase 7 — Integration Testing & User Evaluation
With frontend and backend both running, do a full end-to-end test across multiple emotion scenarios. Test at least one input per major emotion category (sadness, anger, joy, anxiety, calm) and verify that the returned response and playlist match the expected mood.
After integration testing passes, conduct user testing with a minimum of 20 to 30 respondents as recommended for the MIND journal. Use the SUS (System Usability Scale) questionnaire with its 10 standard questions to measure usability. Use the PANAS (Positive and Negative Affect Schedule) scale to measure emotional response before and after using the system. Also collect a satisfaction rating specifically on playlist relevance from 1 to 5.
Collect and analyze the results, compute the average SUS score, and summarize PANAS findings. These results go directly into the Evaluation section of the paper.
